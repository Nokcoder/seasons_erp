# UI Standards

## 1. Keyword Search Bar — Global Standard

Used on: Product Catalogue, Sales Ledger, and all future list/table
views.

### Behavior
- Single search input field
- User types a keyword and hits Enter to commit it as an active filter
  tag
- The committed keyword appears as a dismissible tag/chip below or
  inside the search bar
- The input field clears and is immediately ready for the next keyword
- Multiple keywords can be active simultaneously
- The list filters in real time as the user types the next keyword
  (before hitting Enter) — results narrow to match the partial input
  plus all already-committed keywords
- Removing a tag (clicking its X) immediately updates the results
- All active keyword tags are ANDed together — results must match
  all active keywords, not just one

### Visual
- Active keyword tags are clearly visible and dismissible
- The search input sits inline with the tags, growing the tag list
  naturally as keywords are added
- A Clear All option dismisses all active tags at once

### Scope
- Keyword search spans all text-searchable fields relevant to the
  context (e.g. on the Product Catalogue: brand, variant name, PID,
  SKU, barcode; on the Sales Ledger: sale PID, cashier name, etc.)
- The searchable fields per context are defined in each page's own
  spec file

### Implementation Notes
- Debounce the real-time filtering at 300ms to avoid excessive API
  calls
- Active keyword tags persist in the URL query string so the filtered
  state is shareable and survives a page refresh
- On mobile, the tag list wraps gracefully below the input

## 2. Spreadsheet Import — Upsert Standard

Applies to all XLSX import features across the app.

### PID as anchor
- PID is the universal anchor key for all imports
- If PID exists in the DB → update mode
- If PID does not exist → create mode
- Both modes can coexist in the same import file

### Update mode behavior
- Blank cells = no change to existing value
- Filled cells = update to new value
- Same value as existing = no change (treated as blank)

### Diff modal
- Before applying any updates, system shows a diff modal
- One diff row per affected record
- Left column: current DB values
- Right column: incoming import values
- Unchanged fields greyed out
- Changed fields highlighted

### Confirmation options
- Row by row — user confirms or cancels each record individually
- Bulk confirm all — confirms all pending updates at once
- Both options available in the same modal

### Create mode behavior
- Standard creation — all required fields must be present
- Missing required fields flagged with a clear error per row
- Failed rows do not block successful rows from being created

### Downloadable template
- Every import form provides a downloadable blank XLSX template
- Template headers match the expected import columns exactly
- Template includes a sample row as guidance
- Generated by the backend based on current schema

### Global scope
- Applies to: Product Catalogue, Transfer, Receiving, and all future
  import features

## 3. Supplier SKU Auto-fill — Global Standard

Applies to all supplier link forms across the app.

### Behavior
- When adding a supplier link to a variant, the Supplier SKU field
  pre-fills automatically with the variant's internal SKU value
- User can override manually if the supplier uses a different code
- If variant has no SKU set, field stays blank
- Applies to: Detail page inline form, New Product creation form,
  and XLSX import (blank supplier_sku column copies from sku column
  automatically)

## 4. Data Fetching & Caching — Global Standard

Applies to all data fetching across the app.

### Library
- Use React Query for all server data fetching
- No plain useEffect + fetch patterns for server data
- React context or Zustand for client-only UI state only

### Cache configuration
- Reference data (UOMs, locations, shifts, registers, payment modes,
  roles, categories) — stale time: 10 minutes. Changes rarely,
  serve aggressively from cache.
- Transactional data (sales, transfers, stock levels, ledger entries)
  — stale time: 30 seconds. Refresh in background on window focus.
- User/auth data — stale time: 5 minutes.

### Parallel fetching
- Never waterfall API calls — all independent calls on a page fire
  in parallel
- Use React Query's useQueries or Promise.allSettled
- A slow or failing endpoint must never block other data from
  rendering

### Background refresh
- Show cached data immediately while refreshing in background
- Use a subtle isFetching indicator (thin bar or spinner) — never
  block the UI waiting for a refresh
- On error, show stale data with an error banner — never a blank
  page

### Optimistic updates
- Apply optimistic updates for all predictable create/edit/delete
  operations
- Roll back on error with a clear error message

## 5. Code Splitting & Loading — Global Standard

### Route-level code splitting
- All routes lazy loaded via React.lazy + Suspense
- Established in Batch 1 — enforce on every new page

### Skeleton loaders
- Every page and section that fetches data must show a skeleton
  loader while loading
- Never show a blank area or spinner-only state
- Skeletons approximate the shape of the content they replace

### List performance
- Paginate or virtualize any list expected to exceed 50 rows
- Use cursor-based pagination where possible for large datasets
- Infinite scroll or Load More pattern for history sections
- Never load unbounded lists into the DOM

### Search debounce
- All search inputs debounced at 300ms before firing API calls
- Applies to keyword search, dropdown searches, and any typed filter

### Tab and page transitions
- Navigating between tabs must not re-fetch data that is already
  cached and fresh
- React Query handles this automatically when query keys are
  consistent
- Tab content must render instantly from cache when available

## 6. Product Catalogue Table — Global Standard

### Default variant emphasis
- Default variant row: slightly bolder visual weight + small
  "Default" badge next to variant name
- Non-default sibling rows: slightly muted appearance

### Column visibility
- All columns toggleable via column picker (⚙ button, top right)
- Permanently visible: Brand, Variant Name, PID
- Toggleable: SKU, Product Type, Category, Price, Promo Price,
  Total Stock, Status
- Location columns: separate group in column picker, split into
  Physical Locations and Virtual Locations groups
- Column selection persists in localStorage per user

### Sortable columns
- Click column header to sort ascending, click again for descending
- Sortable by: Brand, Variant Name, PID, SKU, Category

### Price display
- Price and Promo Price as separate columns
- Visual indicator (badge or highlight) when Promo Price is active
- Inherited price on non-default variants shown greyed out

## 7. Product Detail Page — Variant Standards

### Structure
- Routes to /inventory/:variant_id
- Top section: product-level fields (brand, type, description,
  base UOM, categories, status)
- Sibling variants panel: table of all variants under the same
  product, current variant pre-expanded, others collapsible
- Each expanded variant shows: variant fields, barcodes, UOMs
  conversions, bundle components, stock per location, sales history,
  purchase history

### Price inheritance
- Default variant price and promo price are the product-level
  defaults
- Non-default variants with no price override show inherited value
  greyed out with a "Reset to default" option
- Override by entering a different value

### Supplier link inheritance
- Default variant supplier links serve as product-level defaults
- Non-default variants inherit unless explicitly overridden
- Inherited supplier links shown greyed out
- Override per variant when supplier uses different pricing or SKU

### Supplier SKU
- Pre-fills from variant's internal SKU (see Standard §3)

## 8. Bundle Count Mechanic — Global Standard

Applies to all quantity entry fields in stock movement (transfers
and receiving).

### Behavior
- Each variant may have at most one UOM conversion row where
  is_warehouse_bundle = true
- When is_warehouse_bundle conversion exists for a variant:
  - Bundle Count and Qty fields shown side by side
  - User enters Bundle Count → Qty = bundle_count × factor
  - User enters Qty → Bundle Count = ceil(qty / factor)
  - Fractions always round UP to the next whole bundle
- When no is_warehouse_bundle conversion exists for a variant:
  - Bundle Count column hidden for that row
  - Qty field only

### Scope
Transfers (create, detail), Receiving (supplier declaration,
warehouse count), and any future stock movement forms.

## 9. UOM Selling Standard

Applies to all sales encoding interfaces — auditor workstation
and future cashier page.

### Behavior
- When a variant has UOM conversions with prices defined,
  a UOM selector appears on the basket row
- Base unit is always the default selection
- Each priced UOM conversion appears as an additional option
- Selecting a UOM sets the unit price and display quantity
  in that UOM
- Backend receives: variant_id, uom_id (selected), qty in
  selected UOM, unit_price
- Backend resolves base unit deduction:
  qty × conversion factor = base units deducted from stock
- Barcode scan auto-selects the UOM linked to that barcode

### Stock display in catalogue
- Base unit stock shown by default
- For variants with priced UOM conversions, show equivalent
  sellable units per UOM as a tooltip or expandable breakdown
- Equivalent units = floor(base_stock / factor)
- Fractional units are floored — you cannot sell a partial
  pack

### Price resolution hierarchy
1. UOM conversion promo_price (if set and UOM selected)
2. UOM conversion price (if set and UOM selected)
3. Variant promo_price (if set, base unit)
4. Variant price (base unit fallback)

## 10. Numeric Input Field — Value Replacement Standard

Applies to all numeric input fields across the entire app.

### Behavior
- When a user clicks into any numeric input field, the existing
  value is immediately fully selected
- User can start typing immediately to replace the value —
  no manual erasing or double-clicking required
- Applies to: quantity fields, price fields, discount fields,
  cost fields, bundle count fields, and any other numeric input
- Implementation: add onFocus handler that calls
  event.target.select() on every numeric input field

### Scope
All numeric inputs across:
- Auditor workstation basket grid (qty, price, discounts)
- Transfer form line item grid (bundle count, qty)
- Receiving form line item grid (bundle count, qty, unit cost)
- Product Detail page (price, promo price, factor, cost fields)
- Settings forms (payment terms, credit limit)
- Any future numeric input field

### Implementation
```typescript
// Apply to every numeric input
<input
  type="number"
  onFocus={(e) => e.target.select()}
/>
```

Or as a global CSS + JS rule applied to all
`input[type="number"]` elements to avoid having to add
it manually to every field.
